Waarom automa(t/g)isch versies beheren?

Omdat je bij samenwerking wel eens tegelijk in een bestand werkt en rondsturen via e-mail i) heel erg 1971 is en ii) een rotzooi in je mappen geeft. Tegelijkertijd is het wel degelijk handig om oudere versies van je bestanden te bewaren. Daar moet toch een slimme oplossing voor zijn?

> cd teamreport > ls report.html report_20200103_v1.html report_20200104_v2.html report_20200105_v2_joost.html report_20200105_v2_koop.html report_20200105_v2_charley.html report_20200106_v3.html report_20200107_final.html report_20200107.html

Oplossing: Lokaal versiebeheer

Je kunt je een oplossing voorstellen waarbij op jouw computer verschillende versies van bestanden in een systeem opgeslagen worden, dit heet lokaal versiebeheer. Bijvoorbeeld dagelijks automatische backups in timepage maar ook de undo via cmd+z.

Als je per ongeluk een fout maakt kan je altijd een stapje terug naar een eerdere versie. Maar hoe werk je samen?

Oplossing: Gecentraliseerd versiebeheer

Je kunt je ook een oplossing voorstellen waarbij op één server verschillende versies van bestanden opgeslagen worden, dit heet gecentraliseerd versiebeheer. Iedereen verstuurd zijn bestanden naar die ene server en daar blijven alle wijzigingen bewaard. Implementaties van dit principe zijn Subversion en CVS maar ook Git kan op deze manier gebruikt worden.

Als iemand per ongeluk een fout maakt kan iedereen altijd een stapje terug naar een eerdere versie, maar wat als de server omvalt?

Git is een versiebeheersysteem (VCS)

Een VCS helpt je met het uit elkaar houden van verschillende versies van bestanden en mappen. Ook als je met meerdere ontwikkelaars aan hetzelfde project werkt.

Het is een log waarin staat wie, wat, wanneer heeft gedaan (en waarom door middel van commit messages).

  • Er zijn verschillende VCSen:
    • Non-distributed: Subversion, CVS
    • Distributed: Git, Mercurial
